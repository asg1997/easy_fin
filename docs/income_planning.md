# Приход — планирование

## Контекст

- Точка входа: кнопка «Добавить приход» на главной панели → `AddIncomePage` (заглушка).
- Модель `Income` уже есть, но **хранилища и БД нет** — приходы из выписки идут через `BankStatementOperations`, ручные приходы не сохраняются.
- Ближайший аналог по UX и структуре — **начисление по аренде**: один документ (база + дата + счёт) → несколько строк.

---

## Доменная модель

### Документ прихода (заголовок)

Один пользовательский документ = одна операция ввода:

| Поле | Описание |
|------|----------|
| `id` | Уникальный ID документа |
| `baseId` | База |
| `date` | Дата прихода |
| `account` | Касса или банковский р/с |
| `createdAt` | Время создания |

### Строка прихода (позиция документа)

Документ содержит **1..N строк**. Каждая строка — отдельный `Income`:

| Поле | Описание |
|------|----------|
| `sum` | Сумма |
| `incomeSource` | Источник (см. ниже) |
| `note` | Комментарий (опционально) |

### Источники прихода (`IncomeSource`)

**1. От арендатора** (`IncomeSourceFromRenter`)

- Выбор арендатора (и р/с, если несколько).
- В перспективе — **связь с начислениями** (`RenterAssignment`): зачёт оплаты по месяцам, остаток долга.
- На первом этапе достаточно сохранить `renterId` (+ `accountNumber`); логику зачёта — отдельной задачей.

**2. Прочий приход** (`IncomeSourceFromOther`)

- Выбор категории из **настраиваемого справочника** (Настройки).
- Категории: id + название; можно добавлять, переименовывать, архивировать.
- Дополнительно — комментарий к строке (`note`).

> Прочие приходы без привязки к начислениям — самостоятельные операции; категория и `note` — основа для отчётов.

## Архитектура (слои)

```
view/
  pages/add_income_page.dart           — форма: заголовок + список строк + сохранение
  pages/income_categories_page.dart    — справочник категорий (из Настроек)
  providers/...                        — при необходимости

data/
  incomes_storage/                     — CRUD документов и строк
  income_categories_storage/           — CRUD справочника категорий
  drift/models/
    income_documents_table.dart
    income_lines_table.dart
    income_categories_table.dart

models/
  income.dart                          — уже есть, возможно расширить
  income_document.dart                 — заголовок документа (новый)
  income_category.dart                 — категория прочего прихода (новый)
```

### БД (Drift)

```
income_categories
  id, name, is_archived, sort_order, created_at

income_documents
  id, base_id, date, account_type, account_ref, created_at

income_lines
  id, document_id, amount_minor, source_type, renter_id?, account_number?,
  category_id?, note, created_at
```

- `income_lines.category_id` → FK на `income_categories` (RESTRICT при удалении; архив вместо удаления, если категория использовалась).
- `source_type`: `renter` | `other`
- Суммы в minor units (как в `renter_assignments`).
- Миграция `schemaVersion` → 10.
- **Seed при первом запуске** (или в миграции): Кредит, Возврат, Перевод, Прочее — пользователь потом меняет список сам.

### Справочник категорий (Настройки)

- Пункт в `SettingsPage` → экран списка категорий.
- Действия: добавить, переименовать, архивировать/разархивировать.
- В форме прихода в dropdown — только **неархивные** категории.
- Удаление: только если категория ни разу не использовалась; иначе — архив.

### Отображение в таблице документов

По аналогии с начислениями: **одна строка в таблице = один документ**, сумма = сумма всех позиций.

- `DocumentsStorage` — подтянуть ручные приходы.
- `DocumentsTableItem` — расширить идентификатором документа прихода (для удаления/редактирования).
- `DocumentType.income` — уже есть.

### Связь с начислениями (не в v1)

Отдельная сущность `IncomeAllocation` (приход → начисление, сумма) или поле `assignment_id` на строке — проектировать позже, когда появится экран зачёта.

## UX (форма прихода)

1. **Шапка**: база, дата, счёт (касса / банк).
2. **Список строк** (таблица, как в начислениях):
   - тип: «Арендатор» / «Другое»;
   - арендатор + р/с **или** категория прочего прихода;
   - сумма, комментарий.
3. Кнопки: добавить строку, удалить строку, сохранить.
4. **Арендатор не в базе** (из TODO): предупреждение + предложение добавить через `AddRenterDialog` (как в начислениях).
5. Валидация: база и счёт обязательны, ≥1 строка, суммы > 0, без дублей арендатора+р/с в одном документе.

## Решения

### ✅ Один счёт на весь документ

Подтверждено. Счёт (касса или банковский р/с) задаётся в шапке документа и общий для всех строк.

**Примеры:**
- База 1, **касса**, без р/с — 10 арендаторов принесли наличные (10 строк в одном документе).
- База 2, **банк**, р/с `11111111111` — одно перечисление с другого счёта (одна или несколько строк в том же документе).

### ✅ Редактирование в v1

Сохранённый приход можно открыть и изменить (шапка + строки), по аналогии с начислениями.

### ✅ Настраиваемый справочник категорий

Категории прочих приходов — отдельный справочник в **Настройках**:

- добавление и переименование;
- архивирование (скрыть из выбора, сохранить в истории);
- при первой установке — дефолтный набор: Кредит, Возврат, Перевод, Прочее.
