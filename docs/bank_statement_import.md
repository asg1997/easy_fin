# Импорт банковских выписок: ограничения и решения

## Текущее поведение

- XLS → CSV → парсер (Сбер / ВТБ) → сохранение в БД.
- Выписка идентифицируется по `(номер счёта, startDate, endDate)` — повторный импорт **того же периода** обновляет данные, а не создаёт дубликат.
- Операции привязаны к выписке; в таблице документов показываются все операции из загруженных выписок.
- Счёт должен быть привязан к базе, иначе импорт останавливается с запросом создать/выбрать базу.

---

## Ограничения и проблемы

### 1. Нет проверки непрерывности остатков

**Проблема:** входящий остаток новой выписки не сравнивается с исходящим остатком предыдущей.  
Пример: в БД исходящий 100 ₽, в новой выписке входящий 200 ₽ — пропущена выписка или операции, но импорт проходит молча.

**Решение:** перед сохранением сравнивать `initialBalance` новой выписки с `finalBalance` предыдущей по счёту (с учётом хронологии); при расхождении — предупреждение с выбором: отменить / импортировать / отметить разрыв.

**Примечание:** периоды выписок не обязаны стыковаться по календарю (01–15.01 и 20–31.01 — нормально). `initialBalance` новой выписки уже отражает состояние счёта на её `startDate`; пропуск данных ловится расхождением остатков, а не «дырой» в датах.


---

### 2. Нет проверки пересечений периодов

**Проблема:** выписки 01–15.01 и 10–31.01 обе сохраняются → операции 10–15.01 **дублируются** в документах.

**Не проблема:** разрыв календарных дат между выписками (15.01 → 20.01) — штатная ситуация при произвольных периодах выгрузки из банка.

**Решение:** при импорте проверять пересечение: `new.startDate <= existing.endDate && new.endDate >= existing.startDate` для того же счёта; блокировать или требовать подтверждение.

---

### 3. Нет внутренней проверки выписки

**Проблема:** не проверяется, что `initialBalance + Σприходы − Σрасходы ≈ finalBalance`. Ошибка парсинга или битый файл могут пройти незамеченными.

**Решение:** валидация после парсинга; при расхождении — блокировать импорт.

---

### 4. Нет дедупликации операций между выписками

**Проблема:** у операции нет банковского идентификатора (номер документа). Два одинаковых расхода в один день — легитимны; дубли из пересекающихся выписок — нет, но отличить их нельзя.

**Решение:**
- парсить и хранить номер платёжного документа;
- unique-индекс по `(accountNumber, documentNumber)` или fingerprint `(date, amount, счета, note)`;
- запрет пересекающихся периодов как основная защита.

---

### 5. Повторная загрузка той же выписки

**Статус:** частично решено — upsert по периоду, операции перезаписываются.  
**Проблема:** пользователь не видит, что выписка уже была — кажется, что импорт прошёл «с нуля».

**Решение:** сообщение «Выписка за период … уже есть, данные обновлены».

---

### 6. Импорт не по хронологии

**Проблема:** можно загрузить март, потом февраль — проверка остатков усложняется, расхождения труднее заметить.

**Решение:** предупреждение, если новая выписка старше последней сохранённой; опционально — сортировка очереди импорта по `startDate`.

---

### 7. Первая выписка для счёта

**Проблема:** не с чем сравнить входящий остаток; в отчёте по базам при нулевом балансе не подставляется остаток из выписки (см. TODO).

**Решение:** для первой выписки пропускать проверку непрерывности; при `balance == 0` брать `initialBalance` из первой/последней выписки.

---

### 8. Парсер Сбера: несколько листов

**Проблема:** TODO в коде — многостраничные выписки могут парситься не полностью.

**Решение:** обрабатывать все листы XLS; объединять операции в одну выписку.

---

## Рекомендуемый порядок внедрения

| # | Задача | Эффект |
|---|--------|--------|
| 1 | Проверка непрерывности остатков | Ловит пропущенные выписки и операции |
| 2 | Запрет/предупреждение при пересечении периодов | Защита от дублей операций |
| 3 | Внутренняя проверка баланса выписки | Ловит ошибки парсинга |
| 4 | Номер документа + dedup | Точная дедупликация |
| 5 | UI: экран целостности выписок | Визуальный контроль уже загруженных данных |

## Архитектура (целевая)

```
Парсинг → BankStatementImportValidator → [диалог при warnings] → save()
```

Валидатор возвращает список issues с уровнем: `error` (блок) / `warning` (подтверждение) / `info` (уведомление).
